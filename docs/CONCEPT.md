# コンセプト: 触ると図が答えてくれる教科書

中学受験算数の**インタラクティブ教科書**。読む教科書でも、見るだけのアニメでもなく、
**「分からない場所を指でなぞると、そこだけが動いて教えてくれる」**教材。タブレット(iPad)前提。

## コア体験(UXの4原則)

1. **問題ファースト** — まず問題だけを見せて自力で考えさせる。最初から解説を見せない。
2. **つまずいたら「なぞる」** — 分からない場所(線分図の区間、図形の辺、立体の面…)を指でなぞる/タップすると、**その部分に対応するヒントだけ**が動き出す。教えすぎない。
3. **段階ヒント** — 同じ場所を繰り返し求めると深くなる: ①着眼点(どこを見る?) → ②図の操作(動くアニメ) → ③式 → ④全解説(現行のコマ送りシナリオ)。
4. **1画面1問** — タブレット横持ち基準。左に図(操作エリア)、右または下に問題文・ヒント。ボタンは指サイズ(44px+)。

### 体験の具体例(つるかめ算)

- 問題「ツルとカメ合わせて10匹、足26本」→ 白紙の面積図と問題文だけ表示
- 子どもが図の「高さのちがう部分」をなぞる → その帯だけ赤く点滅し「1匹交換すると足は何本増える?」(ヒント①)
- もう一度なぞる → カメ1匹分の交換アニメが1回だけ動く(ヒント②)
- さらに → 式「(26−20)÷2」が出る(ヒント③) → 最後に全解説(既存シナリオ)(ヒント④)

## 既存資産からの進化

| いま(v1) | これから(v2) |
|---|---|
| シナリオ = 一本道のコマ送りアニメ | シナリオ = **部品化されたマイクロアニメの集合**(ヒントグラフ) |
| 見るだけ | 図形要素に**タッチ領域(region)**があり、なぞりに反応 |
| 先生が見せる教材 | 子どもが1人で進める教科書(演習→つまずき→ヒント) |

エンジンの設計思想(シナリオ=データ / エンジン=補間 / verify必須 / 2バックエンド)は**そのまま引き継ぐ**。
ヒントも「小さなシナリオ(steps)」として表現すれば、既存の補間エンジンとビルダーがほぼ再利用できる。

## 技術コンセプト(たたき台)

### 1. region(タッチ領域)スキーマ

ビルダーが描画に加えて「意味のある部品」を宣言する:

```js
regions: [
  { id: "diffBand",  shape: { kind: "rect", x: 7, y: 2, w: 3, h: 2 },  label: "たりない部分" },
  { id: "baseRect",  shape: { kind: "rect", x: 0, y: 0, w: 10, h: 2 }, label: "全部ツルの足" },
]
```

- 2D: ワールド座標の rect / polygon / segment(なぞり判定は線分との距離)
- 3D: 面・辺単位(raycast)。v2初期は「タップ」だけでよい(なぞりは2D優先)

### 2. hint(段階ヒント)スキーマ

```js
hints: {
  diffBand: [
    { level: 1, kind: "ask",   text: "1匹カメに交換すると、足は何本ふえる?" },
    { level: 2, kind: "anime", steps: [ /* 既存step形式のマイクロアニメ */ ] },
    { level: 3, kind: "formula", text: "(26 − 2×10) ÷ (4 − 2)" },
    { level: 4, kind: "scenario", ref: "turtleCrane" },   // 既存の全解説へ
  ],
}
```

**スキーマ(regions/hints)が Claude と CodeX の契約書**。ここを固定すれば、
エンジン側とコンテンツ側を別々のAIが並行開発できる。

### 3. 入力(なぞり)

- PointerEvents でストローク取得 → region との交差判定(サンプリング点の内外判定で十分)
- 誤タッチ対策: 一定長(例: 領域の30%)をなぞったら発火。タップは即発火
- Apple Pencil / 指を区別しない(v2では)

### 4. 演習モード(v2後半)

- 答えの数値入力(テンキーUI) → 正誤判定 → まちがえたら「どこでつまずいた?」をなぞらせる導線
- 進捗記録: **localStorage を進捗・つまずき履歴に限り使用可**(P0で決定)。端末内完結、サーバ送信なし。
  キーは `jukuse:` プレフィクスで統一し、壊れていても本体が動くこと(読み失敗は無視)

### 5. 配布

- 引き続き **HTML1枚 + GitHub Pages**(https://baseball9781-sudo.github.io/jukuse-math/)
- v2で PWA 化(manifest + Service Worker + Three.js同梱)→ iPadホーム画面から起動、オフラインOK
- 対応: iPad Safari 横/縦、Android タブレット

## AI協働体制(Claude Code × CodeX)

- **共通ルール**: リポジトリの `CLAUDE.md`(Claude用)と `AGENTS.md`(CodeX用)に**同じ内容**を書く。
  鉄則は今と同じ——「`npm run verify` を通らないものはコミットしない」「スキーマ(契約)を勝手に変えない」。
- **分担の原則**: *ファイル単位で担当を分ける*。エンジン(`src/engine.js`・`src/render/`・入力系)は
  どちらか一方が担当し、もう一方はビルダー/シナリオ/ヒントデータを量産する。
  エンジンとコンテンツはスキーマ越しにしか依存しないので衝突しない。
- **分担のたたき台**(入れ替え可):
  - CodeX: v2エンジン(タッチ入力・region判定・ヒント再生・タブレットレイアウト)
  - Claude Code: スキーマ設計レビュー、問題コンテンツ量産、検算テスト、教育的な文言
- **統合の場**: GitHub(このリポジトリ)。PR単位でやりとりし、verify + 実機確認を通ったら main へ。
- **定常ループ(ターミナル内で完結)**: ① Claude Code が実装し verify を通す →
  ② `scripts/codex-review.sh` で CodeX がレビュー+直接修正(観点: 算数の正しさ/規約/配色/タブレット表示/文言) →
  ③ Claude Code が `git diff` で CodeX の修正を確認し、verify 再実行 → 問題なければ commit & push。
  CodeX はエンジン(src/engine.js, src/render/)を直接変更しない(提案のみ)ルール。

## 決定事項(P0・2026-07-14確定)

1. **対象**: 小4〜小6・基礎〜標準。文言はやさしく、ヒントは④段階まで手厚く。単元は網羅型。
2. **利用シーン**: **子どもが1人で自学**。ヒントだけで最後まで自走できる文言・導線にする(大人の補足を前提にしない)。
3. **進捗記録**: **localStorage 解禁**(進捗・つまずき履歴のみ。端末内完結、サーバ送信なし)。CLAUDE.md/AGENTS.md の禁止ルールは「進捗記録以外での使用禁止」に改訂済み。
4. **手書きの範囲**: v2は「なぞる/タップ」まで。図への書き込み(補助線)はP4以降の検討課題。
5. **答え入力**: やる。**P3で演習モード**(テンキー入力→正誤判定→つまずき箇所をなぞる導線)を実装。
