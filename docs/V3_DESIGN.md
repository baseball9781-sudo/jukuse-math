# jukuse-math v3 設計方針 —「なぞる」を学習行為にする

> 原文: 雅也(2026-07-17)。Claude注記は「▶注」で挿入。それ以外は原文のまま。

## 0. 一行で
**なぞりを「ヒントを呼び出すトリガー」から「解く行為そのもの」へ移す。**
「答えが書いてある」と「自分で解ける」の間に、**なぞって解く**という中間段階を作る。

---

## 1. v2 の何が違ったか
v2 の「なぞる」は**入力装置**だった。「ここが分からない」と指差す行為であり、
なぞり自体は何も学ばせていない。学びは、その後に出てくるヒントの側にあった。
タップをなぞりに置き換えただけ、とも言える。

v3 の「なぞる」は**学習行為**。薄い式をなぞる＝書く。線分図をなぞる＝図を描く。
なぞり終わったこと自体が「一回解いた」に相当する。

| | v2 | v3 |
|---|---|---|
| なぞりの役割 | ヒントの配達方法 | 練習そのもの |
| 判定したいこと | **どこを**なぞったか | **ちゃんと**なぞれたか |
| 学びの在処 | なぞりの後（ヒント） | なぞりの最中 |

### 教育設計上の位置づけ
これは**フェード付き例題**（faded worked example）の実装。
完成した解答例 → 一部だけ足場を外す → 全部自力、と段階的に支えを抜く。
自学教材の最大の欠陥は「答えを見る」と「自力で解く」の間が断崖であること。そこに階段を作る。

v3 の独自性は、その中間段階を**穴埋め（認知的な足場）ではなく、なぞり（身体的な足場）**にした点。
式を書く手の動き、線分図を引く手の動きが入る。
特に線分図は「描けないから解けない」子が多いため、描く動作を体に通す価値が大きい。

---

## 2. 最大のリスク：なぞりの無思考化
**なぞりは簡単に「無思考の運動」に堕ちる。**
漢字ドリルのなぞりは運筆の習得が目的だが、算数の式のなぞりは運筆が目的ではない。
薄い線をトレースするだけなら、頭を通さずに手が動いて終わる。
**「なぞったのに何も残らない。しかも本人は勉強した気になる」が最悪の結果。**

したがって v3 の設計課題は、一言でいえば：

> **いかにして、手を動かすことが思考を強制するか。**

以下すべての設計判断は、この一点から導かれる。

---

## 3. 中核思想：なぞり順 = 依存順
無思考化を防ぐ主要な手段。**なぞる順序を、解法の論理順序に固定する。**
違う順でなぞっても線は濃くならない。すると**手の動き＝思考の筋道**になる。

### 図の場合
線分図なら「全体 → 部分 → 差」の順にしかなぞれない。
図の**形**をなぞるのではなく、図が**組み上がる順番**をなぞる。

### 式の場合
順序は書き順ではなく**依存順**。

```
① つる + かめ = 10        ← まず分かっていることを書く
② 2 × 10 = 20            ← ①を使って仮定する
③ 26 - 20 = 6            ← ②と問題文の差を取る
④ 6 ÷ (4 - 2) = 3        ← ③を1匹あたりの差で割る
```

▶注(Claude 2026-07-17): 原文の例(足100本)は「合わせて10匹」と矛盾する(解なし)ため、
v2実装済みの「10匹・26本 → カメ3匹」に数値を差し替えた。構造は原文のまま。

②は①をなぞらないと薄いままで触れない。③は②を通らないと現れない。
式を上から順に写すのではなく、**「①があるから②が書ける」という因果の順に手が動く**。

### 図と式の連動（v3 の狙いの中心）
**線分図と式に、同じ順序を共有させる。**
図の「上のすき間」をなぞった瞬間に、式③ `26 - 20 = 6` が薄く現れる。
図の一部と式の一行が 1:1 で対応し、なぞり順を共有する。

> これができると、**図と式が別物だと思っている子（かなり多い）に、
> 同じものの2つの書き方だと体で分からせられる。**

### なぞる単位は「意味のかたまり」
`35 × 2 = 70` を一筆でなぞらせない。「35」「× 2」「= 70」の3ブロックに割り、
ブロックごとに一瞬止まる。この「35って何の数だっけ」と考える隙間が無思考化を防ぐ。
区切りごとに小さく問いかけを出せる（**v2 の4段階ヒント資産がここで生きる**）。

## 4. 操作の三層分離
なぞり・ドラッグ・入力に、それぞれ別の問いを担当させる。

| 操作 | 何を問うているか | ミスの意味 |
|---|---|---|
| **なぞる** | 式の骨格・図の論理 | 解法が分かっていない |
| **ドラッグ** | どの数を使うか | 立式ミス（数の出所） |
| **入力** | いくつになるか | 計算ミス |

```
なぞる:   [つるの数] × 2 + [かめの数] × 4 = 26      ← 骨格は与える
ドラッグ:    ⑩      × 2 +     □      × 4 = ㉖      ← 数の出所を問う
入力:        7      × 2 +     3      × 4 = [  ]    ← 計算だけを問う
```

**これにより「式は合ってるけど計算がずれてるね」が、手書き認識なしで今日言える。**
手書き認識は、これを実際に回してみて「やっぱり自分の字で書かせたい」と
なってから考えれば十分。

### ドラッグの狙いと、その穴
**狙い：** 文章題が解けない子の失敗は、たいてい計算ではなく
**「数字の出所が分かっていない」**こと。目の前の数を意味を考えずに組み合わせる。
ドラッグは「数には出身地がある」を強制的に意識させる。
`20` を式に置くとき、`つるが10羽` の文が光る。

**穴：** 文章題の数字は3〜4個しかないので**総当たりが効く**。
適当に入れて正解ランプが点くまで試せる。これをやられるとドラッグはガチャになる。

**解決：パレットに「導出された数」が増えていく形にする。**

```
最初のパレット:  10   26   2   4
──────────────────────────────
① 2 × 10 = [20]        ← 20 を自分で入力すると
パレット:  10  26  2  4  ⑳    ← 20 がパレットに加わる
──────────────────────────────
② 26 - ⑳ = [6]        ← 20 は①を通らないと存在しない
```

**まだ計算していない数は使えない。** 順序が構造として効き、駒が揃わないので総当たりが破綻する。
これは「なぞり順＝依存順」と完全に同じ思想。図でやったことを、数のパレットでもやっている。

### 認知負荷への対処：操作自体をフェードさせる
小学生に3種類の操作モードを同時に出すと、**算数ではなく UI の学習**になる。

- **導入期** — なぞり + ドラッグ + 入力
- **定着期** — ドラッグを外し、入力のみ
- **仕上げ** — なぞりも外し、白紙で自力

足場を段階的に外すという当初の思想と一貫する。

## 5. 図の指定方法：手書きは入力、コードが出力

### 原則
- **手書き図をそのまま画像として再現する — やらない。**
  なぞり順の固定も、部分ハイライトも、式との連動も全部死ぬ。
  ピクセルの塊には「ここが差」という意味が入っていないため、構造が全部使えなくなる。
- **手書き図を「仕様書」として読み取り、コードが清書して描く — これを採る。**
  意味（この帯は全体、この中かっこは差）を保持したままデータ化する。

手書きの温かみが欲しいだけなら、**描画スタイルを手書き風にする**（線を揺らす、手書き体フォント）。
実装コストは低いのに、見た目は「先生が黒板に描いた図」になる。
**意味を保ったまま、質感だけもらう。**

### プリミティブ語彙
「どれくらい指定できるか」＝**語彙を定義した分だけ、いくらでも**。
中学受験算数の図は、驚くほど語彙が少ない。

| 種類 | プリミティブ |
|---|---|
| 線分図 | 帯／区切り／中かっこ／ラベル／矢印 |
| 面積図 | 長方形／分割／面積ラベル |
| てんびん | 支点／腕／おもり（v2 で既存） |
| ダイヤグラム | 軸／折れ線／交点 |
| 数直線 | 目盛／点／区間 |

この5〜6種で、24問どころか今後の増分もほぼ覆えるはず。
**語彙を作る作業が本体。そこが終われば量産は作業になる。**

### 運用フロー
1. 雅也が手書きで図を描く（何も考えず、いつも授業で描くように）
2. Claude がプリミティブに起こす
3. 語彙にない図が出たら、プリミティブを1つ足す
4. 数問回すと語彙が飽和し、以後 3 が起きなくなる

語彙が決まった後、雅也は**手書きで描くだけ**でよく、データ化は Claude / CodeX の仕事になる。

## 6. データ構造の方向
シナリオに v3 用の `steps` を持たせ、各ステップが以下を持つ：

```
step: {
  id,
  depends_on: [前ステップid],   // なぞり順＝依存順を定義
  figure_trace: {...},          // 図のなぞり対象
  expr_trace:   {...},          // 式のなぞり対象（figure と 1:1 対応）
  palette_add:  [導出される数],  // 正解入力後にパレットへ加わる
  input_slot:   {...},          // 計算を問う欄
  prompt:       "問いかけ"       // v2 の hints を吸収
}
```

- `scenario = データ / engine = 補間` という既存構造の上に乗る。
- なぞり対象の線分データを scenario 側に足す形。

### v2 資産の扱い

| 資産 | 扱い |
|---|---|
| `regions`（急所の座標） | **ほぼ流用可** — なぞり対象の定義に転用 |
| `hints`（4段階の言葉） | **ほぼ流用可** — ステップごとの `prompt` に吸収 |
| なぞり判定ロジック | **作り直し** —「どこを」→「ちゃんと」へ意味が変わる |
| 薄い解説の描画レイヤー | **新規** |
| コマ送り解説（全24問） | **そのまま温存** — 最終段階の全解説として残す |

CodeX が入れた軌跡長ベースの判定は、v3 でも「なぞれたか」の基礎として有効。

▶注(Claude): 「ちゃんとなぞれたか」の判定は (a)経路追従(既存strokeHitScoreのseg判定を流用)
(b)方向(始点→終点の向き) (c)被覆率(経路の何%を通ったか) の3要素に分解して hittest.js に足す。

## 7. 未解決 / 判断待ち
- **なぞりが学習として成立するか** — 未検証。ここが崩れると全部作り直し。
  ▶注(Claude): 実機検証の判定基準を先に決める(例:「2問目を白紙モードで解けるか」)。
- **手書き認識**（自力で書いた行の正誤判定）— 三層分離で代替できるため、当面やらない。
- **フェードの粒度** — 問題ごとか、単元ごとか、習熟度ごとか。
- **iPad 実機でのなぞりの「効き」** — 実機でしか分からない。
- ▶注(Claude): なぞり順ゲートの入口問題 — どこから始めるか分からず固まる子への安全弁として、
  最初のステップだけは常に淡く脈動させる(「分かっていることから書く」自体が教えたい型)。

## 8. 次の一手
**つるかめ算 1問だけで、「線分図を論理順になぞらせる」を作る。**

- 残り19問の quiz 化も PWA 化も、**これが確かめられてからで遅くない。**
  というより、ここが成立しなければ全部作り直しになる。
- 必要な材料：
  1. 対象の問題文（→ v2実装済みの「合わせて10匹・足26本」を使う）
  2. 雅也の手書き線分図を1枚（「これが描ければ解ける」と思っている一枚）
- それを見てプリミティブ語彙を切り、1問分の `steps` を組む。
